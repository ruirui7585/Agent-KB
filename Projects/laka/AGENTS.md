## 1. 文件目的

本文件是 Laka 项目的统一 AI 执行协议。

适用于能够读取本项目目录或已获得本文件内容的 AI 工作环境，包括：

* Codex；
* ChatGPT Work；
* ChatGPT Chat；
* Claude Code；
* Cursor；
* 其他本地或云端 Agent。

Laka 是一个持续运营、持续迭代的长期项目，每周会产生新的产品需求。

所有新需求都必须继承 Laka 已有的：

* 产品背景；
* 公共业务规则；
* 历史产品决策；
* 当前需求状态；
* 真实产品 UI；
* 高保真 HTML 页面基线。

不得要求用户重复提供已经存在于项目文件中的内容。

本文件负责定义“AI 应该如何工作以及到哪里读取信息”，不负责保存完整业务背景、完整历史需求或全部 UI 细节。

## 2. 语言与命名规范

### 2.1 文档语言

项目文档采用以下语言规则：

* 产品背景、业务规则、PRD、决策原因、验收标准：以中文为主；
* 文件名、目录名、状态值、字段名、代码标识、接口字段：使用英文；
* App 前台界面文案：默认使用英文；
* 必要时在英文术语首次出现时补充中文解释；
* 不为了“方便模型理解”强制把所有文档翻译成英文。

AI 必须能够理解中文业务规则，不得因为文档以中文编写而降低执行完整性。

### 2.2 文件与目录命名

文件名和目录名必须：

* 使用小写英文；
* 单词之间使用连字符 `-`；
* 不使用空格；
* 不使用中文目录名；
* 不使用 `new`、`latest`、`final-final` 等无明确版本意义的名称。

固定公共文件保留约定的大写名称：

```text
AGENTS.md
PROJECT_CONTEXT.md
DECISIONS.md
REQUIREMENTS_INDEX.md
CHATGPT_CONTEXT.md
UI_SYSTEM.md
SCREEN_SPEC.md
README.md
PRD.md
```

## 3. 项目根目录识别

Laka 项目会在多台电脑上使用，本地绝对路径可能不同。

当前已知合法路径：

```text
家庭电脑：
/Users/judy/Agent/Projects/laka

公司电脑：
/Users/shilv/Agent-Workspace/Projects/laka
```

以上路径属于同一个 Laka 项目在不同设备上的本地副本。

不得将其中任意一个绝对路径写成唯一有效路径。

### 3.1 PROJECT_ROOT

执行任务时，将当前工作环境中已经打开的 Laka 根目录定义为：

```text
PROJECT_ROOT
```

项目根目录应满足：

* 当前目录或其父级目录名为 `laka`；
* 根目录存在 `AGENTS.md`；
* 根目录存在或预期存在 `PROJECT_CONTEXT.md`；
* 根目录存在或预期存在 `requirements/`；
* 根目录存在或预期存在 `ui/`。

在终端环境中，应优先通过以下方式确认：

```bash
pwd
```

不得仅根据用户名、历史路径或预设绝对路径判断当前环境。

### 3.2 相对路径原则

所有项目内部引用必须优先使用相对于 `PROJECT_ROOT` 的路径。

正确示例：

```text
PROJECT_CONTEXT.md
DECISIONS.md
REQUIREMENTS_INDEX.md
ui/UI_SYSTEM.md
ui/screens/im-chat/baseline.html
requirements/active/
requirements/archive/
```

禁止在项目正式文件中硬编码：

```text
/Users/judy/...
/Users/shilv/...
```

只有执行本机命令或用户明确指定外部文件时，才可以临时使用绝对路径。

不得因为某个历史绝对路径不存在，就在该路径下自动创建另一份 Laka 项目。

## 4. 环境读取边界

### 4.1 本地 Agent

当 Codex、Work 或其他本地 Agent 已经打开 `PROJECT_ROOT` 时，应读取并遵循本文件。

### 4.2 网页版或普通 ChatGPT Chat

如果当前聊天无法访问本地文件系统：

* 不得声称已经读取本地路径；
* 应使用用户上传或添加到项目中的文件；
* 应优先读取 `CHATGPT_CONTEXT.md` 和当前需求文件；
* 不得仅凭本地路径名称推测文件内容。

`CHATGPT_CONTEXT.md` 是提供给 ChatGPT 项目的上下文汇总镜像，不是独立事实源。

## 6. 文件职责

### `AGENTS.md`

定义：

* 执行流程；
* 文件读取顺序；
* 修改权限；
* 交付标准；
* 验证和归档规则。

### `PROJECT_CONTEXT.md`

保存所有需求默认继承的长期公共事实，例如：

* 产品定位；
* 用户角色；
* 产品模块；
* 商业模式；
* 平台级业务规则；
* 核心术语；
* 重要业务数据口径。

### `DECISIONS.md`

保存经过用户确认、未来仍可能被复用或错误推翻的重要产品决策及原因。

不用于保存完整历史需求。

### `REQUIREMENTS_INDEX.md`

作为需求总台账，保存：

* 当前需求；
* 需求优先级；
* 文档状态；
* 产品状态；
* 关联页面；
* 当前目录；
* 归档目录；
* 阻塞和待确认问题。

### `CHATGPT_CONTEXT.md`

汇总以下内容，供无法直接读取本地目录的 ChatGPT 项目使用：

* `PROJECT_CONTEXT.md` 的核心信息；
* 当前需求索引；
* 当前有效的重要决策；
* UI 公共规则和核心页面索引。

不得只更新 `CHATGPT_CONTEXT.md` 而不更新正式源文件。

### `ui/`

保存 Laka 当前产品的真实 UI 基线、页面说明和公共 UI 规则。

### `requirements/`

保存每次正式需求的完整增量资料和最终交付物。

## 7. 信息优先级

当信息发生冲突时，按以下优先级处理：

1. 用户在当前任务中的明确指令；
2. 当前需求最新有效的 `README.md` 和 `PRD.md`；
3. 用户明确提供并说明为当前版本的产品截图或 HTML；
4. 对应页面的 `baseline.html`、`reference.png` 和 `SCREEN_SPEC.md`；
5. `PROJECT_CONTEXT.md`；
6. `DECISIONS.md` 中状态为 `Confirmed` 的决策；
7. `REQUIREMENTS_INDEX.md`；
8. 历史需求和归档文件；
9. AI 自身推测。

发现冲突时：

* 使用高优先级信息；
* 不得静默混用两套规则；
* 在交付说明中指出冲突；
* 不得把一次临时修改自动升级成公共规则；
* 不得用归档需求覆盖当前版本。

## 8. 任务开始前的读取规则

### 8.1 所有正式需求必须读取

```text
AGENTS.md
PROJECT_CONTEXT.md
REQUIREMENTS_INDEX.md
```

并根据当前任务在 `DECISIONS.md` 中检索相关决策。

如果需求目录已经存在，还必须读取：

```text
requirements/active/<requirement>/README.md
requirements/active/<requirement>/PRD.md
```

### 8.2 UI 或高保真 HTML 任务必须额外读取

```text
ui/UI_SYSTEM.md
ui/references/reference-index.md
assets/asset-manifest.md
ui/screens/<screen>/baseline.html
ui/screens/<screen>/reference.png
ui/screens/<screen>/SCREEN_SPEC.md
```

其中：

* `ui/references/reference-index.md` 用于识别正式版截图、主参考、辅助状态、重复截图和禁止继承的问题；
* `assets/asset-manifest.md` 用于识别正式素材、线上截图参考、临时素材、缺失正式素材和禁止复用素材。

不得只查看页面截图而跳过截图索引。

涉及宫殿背景、底部导航、货币、VIP、Relationship、徽章或其他品牌素材时，不得跳过素材清单。

如果找不到对应页面，应依次检索：

```text
ui/screens/
requirements/active/
requirements/archive/
assets/
```

### 8.3 历史需求读取条件

仅在以下情况读取归档需求：

* 用户明确要求参考历史需求；
* 当前需求属于历史需求二期；
* 当前需求与历史功能高度相似；
* 需要追溯某个重要决策；
* 当前有效文件明确引用历史需求。

归档文件不得默认视为当前规则。

## 9. 正式需求判断

满足以下任一条件时，视为正式需求：

* 需要输出正式 PRD；
* 需要输出高保真 HTML；
* 需要新增或调整业务规则；
* 需要设计前端交互；
* 需要设计后台功能；
* 需要设计接口、状态或数据逻辑；
* 需要提供开发或测试验收标准；
* 需要交付给设计、开发、测试或运营执行。

以下情况默认不创建正式需求目录：

* 单句文案优化；
* 单个概念解释；
* 临时问题咨询；
* 尚未形成交付目标的探索讨论；
* 纯头脑风暴。

当讨论进入正式方案或交付阶段时，再创建需求目录。

## 10. 正式需求目录

正式需求创建在：

```text
requirements/active/
```

目录命名格式：

```text
YYYYMMDD-english-requirement-name
```

示例：

```text
requirements/active/20260712-wishlist-gift/
requirements/active/20260715-message-request/
```

前端用户可见需求的最低结构：

```text
requirements/active/<requirement>/
├── README.md
├── PRD.md
├── prototype.html
└── assets/
```

没有专属资源时，可以不创建 `assets/`。

后台、数据、接口或纯规则需求可以不创建 `prototype.html`。

## 11. 修改前执行规则

开始修改前，必须先输出简短计划，包括：

* 当前任务目标；
* 需求类型；
* 涉及的业务模块；
* 涉及的页面；
* 准备读取的文件；
* 准备新增或修改的文件；
* 是否会修改公共上下文；
* 是否会修改 UI 基线。

输出计划后直接继续执行，不需要等待用户再次确认。

以下情况必须先获得明确授权：

* 删除正式文件；
* 批量移动或重构目录；
* 覆盖无法恢复的内容；
* 修改与当前需求无关的 UI；
* 将 AI 建议写入公共业务规则；
* 修改未经用户确认的平台级事实；
* 操作其他项目或项目外关键文件。

用户在当前任务中已经明确授权时，不重复询问。

## 12. UI 事实源

### 12.1 核心原则

已有产品页面必须基于实际 Laka 产品页面进行增量修改。

不得凭空重新设计已有核心页面。

已有核心页面包括但不限于：

* 1v1 IM 聊天页面；
* 首页或 For You 页面；
* 用户主页；
* 会话列表；
* 私密内容页面；
* 守护关系页面；
* CP 关系页面；
* 充值或收银台页面；
* 语音房页面。

### 12.2 UI 事实源优先级

1. 用户当前明确提供的真实产品截图或 HTML；
2. `ui/screens/<screen>/baseline.html`；
3. `ui/screens/<screen>/reference.png`；
4. `ui/screens/<screen>/SCREEN_SPEC.md`；
5. 当前需求中的已确认原型；
6. 历史需求中的已确认原型；
7. `ui/UI_SYSTEM.md`。

### 12.3 reference.png 定义

`reference.png` 必须是：

* 当前真实产品截图；
* 或用户明确确认可以作为当前产品视觉依据的截图。

不得使用 AI 自行生成的原型截图冒充真实产品截图。

### 12.4 baseline.html 定义

`baseline.html` 表示当前可复用的产品 UI 基线。

它不是：

* 任意探索方案；
* 第一版需求原型；
* 尚未确认的未来版本；
* 临时活动页面。

## 13. AI 自行创建 UI 的边界

AI 可以自行创建：

* 当前产品不存在的全新页面；
* 新功能组件；
* 新弹窗；
* 新浮层；
* 新 Banner；
* 新增交互状态；
* 探索阶段的临时方案。

自行创建时必须：

* 遵循 `ui/UI_SYSTEM.md`；
* 延续 Laka 的既有视觉风格；
* 保持相似的信息密度；
* 使用当前产品已有的组件语言；
* 不改变整体设计风格；
* 明确标记未经确认的假设；
* 不得声称自行创建的页面是当前线上真实页面。

如果新功能需要放在聊天页、首页、主页等已有基础页面中：

* 必须使用实际产品基础页面；
* 不得从空白画布重新构建一个近似页面；
* 不得因为缺少局部信息而重做整个页面。

---

## 14. 缺少实际产品页面时

如果需要修改已有核心页面，但找不到当前截图、HTML 或有效基线：

1. 先搜索全部 UI 文件；
2. 搜索当前需求；
3. 搜索归档需求；
4. 搜索项目已有图片和资源；
5. 可以先设计局部新增组件；
6. 局部方案必须标记为 `Provisional UI`；
7. 不得将临时方案升级为 `baseline.html`；
8. 最终交付前必须获得实际产品页面依据。

不得用“AI 自己认为合理”的页面替代已有产品事实。

---

## 15. UI 增量修改流程

已有页面的需求采用：

```text
读取真实 UI 基线
→ 复制到当前需求目录
→ 修改 prototype.html
→ 保留未涉及区域
→ 完成交互与验证
→ 根据产品状态决定是否升级 baseline
```

除非用户明确要求，否则不得：

* 删除原有 UI；
* 移动原有模块；
* 改变已有组件顺序；
* 修改与当前需求无关的交互；
* 替换真实产品素材；
* 重做整个页面；
* 简化其他已有功能；
* 擅自修改背景、导航、资料卡或输入区域；
* 直接覆盖公共 `baseline.html`。

新增 UI 必须说明：

* 展示位置；
* 层级；
* 尺寸；
* 展示条件；
* 触发条件；
* 点击行为；
* 状态变化；
* 与原有 UI 的关系；
* 是否遮挡现有内容。

## 19. 竞品截图参考要求

创建或修改高保真 HTML 前，先判断当前需求是否已有明确、有效的 UI 参考。

以下情况无需搜索竞品：

* 用户当前提供了明确的截图、原型图或 HTML；
* 蓝湖存在当前有效设计稿；
* 对应页面已有有效的 `baseline.html`、`reference.png`、`SCREEN_SPEC.md`；
* 当前需求已有用户确认的原型；
* 现有参考足以确定布局、展示位置、信息层级和主要交互状态。

以下情况必须搜索竞品：

* 没有参考图，只有文字需求；
* 现有参考无法支持高保真实现；
* 新增组件、弹窗、付费状态或异常状态没有参考；
* 属于 Laka 尚未出现的全新页面或交互；
* 用户明确要求进行竞品调研。

如果只有局部 UI 缺少参考，仅搜索缺失的组件或状态，无需重新调研整个页面。

触发搜索后：

1. 至少查找 3 个相关产品；
2. 将截图、预览或有效链接发给用户查看；
3. 简要说明可参考部分和与 Laka 的差异；
4. 用户看过后再继续高保真 HTML。

竞品只用于补充布局、信息层级和交互细节，不得覆盖 Laka 真实 UI 基线、已确认设计、业务规则或品牌风格。

---

## 20. 高保真 HTML 规范

`prototype.html` 必须：

* 是完整 HTML 文件；
* 不拆成零散代码片段；
* 默认使用单文件结构；
* CSS 内联；
* JavaScript 内联；
* 包含 viewport 配置；
* 默认按 iPhone 16 逻辑尺寸设计；
* 推荐基础画布为 `393 × 852`；
* 能直接在浏览器打开；
* 不依赖复杂构建流程；
* 不出现空白页；
* 不依赖失效的关键资源；
* 所有需求交互均可操作；
* 所有关键状态均有可验证入口；
* 保留用户未要求修改的现有 UI；
* App 前台文案默认使用英文；
* 产品说明和 PRD 默认使用中文。

如果当前 `baseline.html` 已有稳定结构，应延续原结构，不得为个人偏好强制重写。

---

## 21. 图片与资源规则

资源使用优先级：

1. 项目内真实产品资源；
2. 当前页面已有资源；
3. `assets/` 公共资源；
4. 当前需求 `assets/`；
5. 稳定的网络资源。

允许原型阶段使用稳定网络链接。

但必须：

* 核心图标和背景优先本地化；
* 不使用明显不稳定的关键链接；
* 不因外链失败导致空白页；
* 不擅自替换用户提供的真实素材；
* 升级公共基线时尽可能保存关键资源到项目内；
* 在交付说明中标记关键外部资源。

## 22. 公共文件更新规则

### 22.1 REQUIREMENTS_INDEX.md

可以自动更新：

* 新需求创建；
* 文档状态；
* 产品状态；
* 当前阶段；
* 关联页面；
* 当前目录；
* 归档目录；
* 阻塞项。

### 22.2 DECISIONS.md

只有满足以下条件才写入：

* 用户明确确认；
* 未来可能再次使用；
* 容易被后续 Agent 错误推翻；
* 存在重要方案取舍；
* 影响收入、成本、合规、体验或多个模块。

不得写入：

* AI 未确认建议；
* 临时 UI 调整；
* 单次文案修改；
* 完整历史 PRD；
* 尚在讨论的选项。

### 22.3 PROJECT_CONTEXT.md

只有在形成已确认、跨需求长期有效的平台规则时更新。

单个需求自己的特殊规则不得写入。

### 22.4 UI_SYSTEM.md

只有形成全局 UI 规则时更新。

单页面规则写入对应 `SCREEN_SPEC.md`。

### 22.5 baseline.html

只有满足以下条件时才升级：

* 方案已确认；
* 页面变化是长期产品能力；
* 该版本已经成为正式产品基线，或用户明确要求升级；
* 不存在影响页面成立的关键阻塞。

需求文档已经归档，但产品尚未采用该页面时，不得升级公共 `baseline.html`。

### 22.6 SCREEN_SPEC.md

页面长期状态、固定组件顺序、权限、交互和不可修改区域发生正式变化时更新。

### 22.7 CHATGPT_CONTEXT.md

当正式源文件发生变化后同步更新。

不得只修改汇总镜像。

## 23. 需求完成与归档

用户选择的默认规则是：

> PRD 和应交付的高保真 HTML 完成后自动归档。

但“完成”必须同时满足质量条件。

### 23.1 前端需求归档条件

1. `README.md` 已完成；
2. `PRD.md` 已完成；
3. `prototype.html` 已完成；
4. PRD 与 HTML 一致；
5. HTML 已完成可执行范围内的检查；
6. 验收标准已覆盖；
7. 不存在阻止交付的未解决问题；
8. 用户未明确表示仍在讨论或当前不是最终版；
9. `REQUIREMENTS_INDEX.md` 已更新；
10. 公共知识提炼已经完成。

不得因为第一版 PRD 和第一版 HTML 已生成就立即归档。

### 23.2 非前端需求归档条件

后台、数据、接口或纯规则需求，在声明的正式交付物完成并通过检查后归档，不强制要求 HTML。

### 23.3 归档含义

归档表示：

* 当前需求文档包已经完成；
* 当前需求不再占用 `active/` 工作区。

归档不必然表示：

* 已开发；
* 已上线；
* 已成为当前产品 UI；
* 已升级公共基线。

产品实际状态必须单独记录。

### 23.4 归档路径

```text
requirements/archive/YYYY/YYYYMMDD-english-requirement-name/
```

归档时：

* 移动完整需求目录；
* 保留 PRD、HTML 和资源；
* 更新 `REQUIREMENTS_INDEX.md`；
* 不删除正式资料；
* 不把完整需求复制进 `DECISIONS.md`；
* 不把归档文件默认视为当前规则。

跨需求长期有效的确认规则写入：

```text
PROJECT_CONTEXT.md
```

未来可能复用或错误推翻的确认决策写入：

```text

ui/screens/<screen>/baseline.html
```

只有取得真实产品截图后，才更新：

```text
ui/screens/<screen>/reference.png
```

不得伪造真实产品截图。

## 25. HTML 验证

交付前尽可能检查：

### 文件

* 文件存在且非空；
* CSS 生效；
* JavaScript 可加载；
* 资源路径有效；
* 页面可直接打开。

### 页面

* 无空白页；
* 移动端尺寸合理；
* 主要区域完整；
* 无关键元素溢出；
* 无重要内容被遮挡；
* 视觉风格未偏离 Laka。

### 交互

* 关键按钮可点击；
* 状态切换有效；
* 弹窗可打开和关闭；
* 返回操作有效；
* 关键入口不会点击无响应。

### 基线

* 未删除无关组件；
* 未修改无关 UI；
* 未改变用户未要求调整的组件顺序；
* 新功能位置符合需求；
* 新功能不遮挡主要内容。

### 逻辑

* PRD 与 HTML 一致；
* 用户角色一致；
* 状态一致；
* 权限一致；
* 边界条件一致；
* 界面文案与交互一致。

### 右侧预览自动打开

每次完成 HTML 的新建、修改、优化或修复后，必须自动将当前有效 HTML 在右侧内置浏览器中打开，供用户直接查看，不需要等待用户再次确认。

执行要求：

1. 必须在文件保存并完成当前可执行的验证后，再打开最终有效版本；
2. 如果右侧浏览器已经打开同一个 HTML，应自动刷新该页面；
3. 如果尚未打开，应使用当前有效文件路径或本地预览 URL 新建页面，并将右侧浏览器显示给用户；
4. 不得只在回复中提供链接而不执行打开或刷新操作；
5. 不得打开旧版本、归档版本、临时副本或与当前需求无关的 HTML；
6. 如果页面依赖本地服务，应先启动可用的本地预览服务，再打开对应 URL，并保持服务运行以便用户查看；
7. 最终回复必须说明右侧页面是“已打开”还是“已刷新”，并同时提供可点击的文件路径或预览 URL；
8. 如果当前环境不支持右侧内置浏览器，或打开操作失败，不得声称已经打开；必须说明具体限制，并提供当前有效 HTML 的可点击路径。

如果当前环境不能实际运行 HTML：

* 不得声称已经完成浏览器运行验证；
* 应执行静态检查；
* 明确说明尚未实际验证的项目。

## 26. PRD 验证

交付前检查：

* 是否说明业务问题；
* 是否明确目标和非目标；
* 是否区分消费侧和供给侧；
* 是否覆盖主客态；
* 是否覆盖状态变化；
* 是否覆盖权限；
* 是否覆盖前端、服务端和后台；
* 是否覆盖异常和边界；
* 是否覆盖版本兼容；
* 是否定义数据或埋点；
* 是否包含可测试的验收标准；
* 是否区分已确认和待确认；
* 是否与公共业务规则冲突；
* 是否遗漏受影响模块。

## 27. AI 假设规则

AI 可以对非关键细节采用合理默认值继续执行，但必须：

* 优先使用项目已有上下文；
* 采用风险最低的方案；
* 明确标记假设；
* 不把假设写成确认事实；
* 不把假设自动写入公共文件；
* 不用假设覆盖真实产品 UI。

以下内容不得静默假设：

* 收费；
* 主播收益；
* 用户权限；
* 合规；
* 数据删除；
* 状态机；
* 历史数据；
* 版本兼容；
* 核心页面结构；
* 永久公共规则。

---

## 28. 禁止行为

任何执行者不得：

* 把某一个设备绝对路径视为唯一项目路径；
* 在不存在的历史路径下创建新的 Laka 副本；
* 修改其他项目；
* 删除用户未要求删除的文件；
* 删除用户未要求删除的 UI；
* 擅自重构整个页面；
* 使用归档文件覆盖当前版本；
* 把第一版草稿称为最终版本；
* 把 AI 建议称为确认规则；
* 把临时 UI 升级为公共基线；
* 在没有真实依据时伪造已有核心页面；
* 把 AI 原型截图当作真实产品截图；
* 重复询问项目文件中已经明确的问题；
* 创建多个含义相同的最终版文件；
* 在多个公共文件中重复维护同一事实；
* 声称完成了实际未执行的验证；
* 将完整历史需求塞入 `AGENTS.md`；
* 把 `CHATGPT_CONTEXT.md` 当作独立事实源。

## 30. 标准工作闭环

```text
识别 PROJECT_ROOT
→ 读取公共上下文
→ 检索相关决策和需求
→ 定位真实产品 UI
→ 创建或更新需求目录
→ 输出执行计划
→ 完成 PRD
→ 基于真实页面完成高保真 HTML
→ 验证 PRD 与 HTML
→ 提炼公共规则和重要决策
→ 根据产品状态决定是否升级 UI 基线
→ 更新需求索引
→ 满足条件后归档需求
```

聊天讨论是需求形成过程。

正式文件才是项目事实、交付结果和后续执行依据。
