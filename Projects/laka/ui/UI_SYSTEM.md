# Laka UI System

> 状态：生效中
> 产品基线：Laka iOS 1.4.7 正式版
> 截图设备：PM 提供的 iPhone 17 Pro
> 主验收视口：440 x 956 CSS px
> 最后确认：2026-07-13

## 1. 文件目的

本文件是所有 Laka UI 任务的全局执行合同，负责回答：

1. 哪个信息源有最终解释权；
2. 哪些产品主风格必须保持稳定；
3. Agent 什么时候可以直接执行，什么时候必须询问；
4. 哪些素材可以复用、临时生成或必须等待正式文件；
5. UI 结果按什么标准验收。

页面结构、页面状态和局部例外写入对应的
`ui/screens/<screen>/SCREEN_SPEC.md`；截图含义写入
`ui/references/reference-index.md`；素材来源写入
`assets/asset-manifest.md`。

## 2. 不可偏离的产品气质

Laka 必须具有：

- 亲密；
- 年轻；
- 娱乐感；
- 关系感。

Laka 不能具有：

- 低俗或性暗示廉价感；
- 幼稚、玩具化；
- 低成本、一次性活动页质感；
- 严肃婚恋、相亲或婚介感。

这不是要求所有页面都做成粉色或紫色。真正的约束是：人物、对话、礼物、房间、身份、关系进度必须成为视觉中心，同时保持成年社交产品的可信度。

## 3. 信息源优先级

### 3.1 业务真值与视觉真值分开判断

业务真值回答“功能、数值、权限和行为是什么”；视觉真值回答“页面如何组织和呈现”。两者必须分别确认，不能用截图中的业务数值覆盖当前配置，也不能用纯业务 PRD 默认为页面改版。

业务优先级：

1. 当前已确认的需求或 PRD；
2. PM 明确确认的当前产品行为；
3. Laka iOS 1.4.7 正式版行为；
4. 历史需求和历史设计。

视觉优先级：

1. 当前需求明确确认采用的蓝湖设计；
2. 同页面、同状态的 Laka iOS 1.4.7 真实产品截图；
3. 蓝湖中的正式素材、尺寸、颜色和组件细节；
4. 对应页面的 `SCREEN_SPEC.md`；
5. 本全局文件；
6. 被明确标注为“推断”的临时决定。

只有业务需求、没有明确提出视觉改版时，必须继承当前 UI，只做实现业务逻辑所必需的最小视觉调整。

### 3.2 蓝湖使用规则

蓝湖是正式设计和素材来源，但蓝湖中的每张设计稿不一定仍然生效。

- 不能仅因为设计位于第 7、8、9 期，就自动认定它是当前版本；
- 一个需求只有一个明确匹配的生效设计时，可以直接采用；
- 出现多个互相冲突的候选稿时，列出分组和页面名，让 PM 选择；
- 找不到当前设计时，继承正式版产品截图；
- 禁止默默混合多个冲突版本的局部细节；
- 每个导入素材都必须记录蓝湖项目、分组、页面名和获取日期。

当前蓝湖项目：

- 项目：`Laka 7-9期后`
- 地址：`https://lanhuapp.com/web/#/item/project/stage?tid=edb6ff63-e81e-4474-aef0-9c0a542f7922&pid=80169c5f-183c-440f-83a5-5e10f1427f60`

### 3.3 正式版截图使用规则

真实产品截图用于确定当前已上线的构图、结构和状态。

- 保留页面结构、信息层级和产品识别度；
- 不继承明显问题，包括遮挡、裁切、错误日期、间距失衡、错误文案和重复内容；
- 已知问题必须在截图索引或页面规范中标注“不继承”；
- 禁止从截图中裁出真实用户照片作为可复用素材；
- `reference.png` 必须一直是真实产品截图，不能被蓝湖图或生成图替换。

## 4. UI 意图确认机制

每个进行中的需求，在实现前必须在该需求的 `README.md` 或 `PRD.md` 中记录 UI 意图卡：

```text
UI Intent Card
- 业务目标：
- 入口和影响页面：
- 用户角色和平台：
- 要改变什么：
- 哪些区域必须保持不变：
- 必须覆盖的状态：
- 当前产品截图：
- 蓝湖来源（如有）：
- 素材状态：
- 待确认问题：
- 确认状态：
```

执行规则：

- 明确且局部的文案、图标、显隐、顺序或范围调整：复述理解后直接执行；
- 新页面、新布局、新主题、商业化流程、角色差异、权限或跨页面行为：得到 PM 确认后再执行；
- 影响范围不清楚或正式来源冲突：先问，不自行发挥。

本规范完成后，不再向 PM 询问常规颜色、圆角、间距、字体和按钮样式。只询问业务目标、入口、行为变化、角色、权限、状态和正式来源冲突。

## 5. 产品主框架

### 5.1 固定金色宫殿主题

以下五个一级 Tab 默认固定使用金色宫殿主题：

- Home；
- Moment；
- Party；
- Message；
- Mine。

固定主题包括：

- 顶部导航区域使用明亮金色宫殿、拱门背景；
- 宫殿背景向白色内容区柔和过渡，确保信息可读；
- 底部五项导航使用同一宫殿视觉语言；
- 激活文字使用深梅紫，激活指示使用鲜明紫色；
- Tab 使用正式插画素材，不能替换为普通线性图标。

宫殿背景属于品牌位图素材。禁止用 CSS 渐变、SVG 几何或 AI 图近似重建。缺少正式素材时必须停止并询问。

### 5.2 可独立主题的功能页

以下功能可以在正式截图或蓝湖支持下使用独立主题：

- Recharge / Checkout；
- VIP；
- Relationship / CP；
- BD Center；
- Tasks；
- Voice Room；
- Shop、Wealth、Charm、Diamonds Center。

独立主题仍需保持 Laka 语义：成年社交、信息清晰、奖励有表现力、导航一致、装饰有价值感，不能廉价。

## 6. 男女端与平台

### 6.1 男女端

大部分 UI 共用同一套系统，不建立两套设计语言。

以下功能全局共用、不因性别改变视觉系统：BD、Relationship、VIP、全局导航、素材、弹窗、商业化组件和系统状态。

允许由业务规则造成差异的内容包括：

- 推荐对象和展示性别；
- 任务定义和奖励币种；
- Income、Agency、Chat Center 等入口；
- 权限、匹配规则和变现行为。

需求未说明男女差异时，只改变内容或可执行操作，不改变组件结构和主视觉。

### 6.2 平台

- iOS 1.4.7 截图是当前视觉基线；
- 主结果必须在 440 x 956 验收；
- 其他常见 iOS、Android 尺寸必须自然重排，不能整屏机械缩放；
- 必须处理安全区、系统状态栏、键盘高度和底部手势区；
- Android 可以适配系统栏和字体指标，但不能形成另一套视觉语言。

## 7. 全局基础规范

以下数值来自已提供截图，是当前实现工作值。蓝湖正式量取值或页面规范可以覆盖它们。

### 7.1 颜色角色

| 角色 | 工作值 | 用途 |
| --- | --- | --- |
| 主文字 | `#2D2D32` | 主标题、正文、主要图标 |
| 次文字 | `#8E8E96` | 时间、说明、未激活标签 |
| 禁用 | `#C9C7CF` | 禁用文字和线性图标 |
| 页面底色 | `#F6F6F8` | 中性页面背景 |
| 表面 | `#FFFFFF` | 卡片、列表、弹窗 |
| 分隔线 | `#ECEBF0` | 低干扰分隔 |
| Laka 紫 | `#603CFF` | 激活 Tab、主要操作 |
| 深梅紫 | `#35104F` | 宫殿主题激活文字和深色图标 |
| 关系粉 | `#FF4FAE` | 亲密度、爱心、未读强调 |
| 奖励橙 | `#FF9E2F` | 金币、价格、暖色奖励 |
| 在线薄荷 | `#3FDEC2` | 在线状态 |
| 消息蓝 | `#55BDF4` | 聊天入口、信息型社交操作 |

颜色规则：

- 运营工具页不能被一种色相全面占据；
- 紫色表达交互品牌，粉色表达关系，橙金表达奖励和商业化，薄荷表达在线，蓝色表达沟通；
- 装饰背景上的正文必须保持中性色和足够对比度；
- 没有截图或蓝湖依据时，不能随意增加装饰渐变。

### 7.2 字体与排版

- 使用平台系统无衬线字体；
- 支持阿拉伯语和 RTL/LTR 混排；
- 字间距不能为负值；
- 字号不能随视口宽度缩放；
- 使用克制的层级：

| 角色 | 字号 | 字重 |
| --- | --- | --- |
| 导航标题 | 17-18 | Semibold |
| 一级 Tab / 页面主标题 | 20-22 | Bold |
| 分区标题 | 18-20 | Semibold / Bold |
| 卡片标题 | 16-18 | Semibold |
| 正文 | 15-17 | Regular |
| 元信息 | 12-14 | Regular |
| 徽章 | 11-13 | Medium / Semibold |

混排规则：

- 文字方向跟随内容，不只跟随应用语言；
- 用户名、国旗、徽章和数值不能互相遮挡；
- 数字、日期、ID 和货币保持完整视觉分组；
- 截断时保留最能识别身份或业务含义的部分。

### 7.3 间距与尺寸

- 基础网格：4 px；
- 页面横向边距：12-16 px；
- 卡片内边距：12-16 px；
- 关联元素间距：8-12 px；
- 分区间距：16-24 px；
- 常规控件高度：44-48 px；
- 强主按钮高度：52-56 px；
- 最小点击区域：44 x 44 px；
- 小标签圆角：8-12 px，语义适合时使用胶囊；
- 卡片圆角：12-16 px；
- 弹窗和 Bottom Sheet 顶部圆角：20-28 px。

棋盘、座位、头像格、商品网格、计数器和导航栏必须使用稳定尺寸、网格轨道、宽高比或 min/max 约束。动态内容不能推动相邻控件变形。

### 7.4 层级与阴影

Laka 主要依靠对比和图片建立层级，不依赖厚重阴影。

- 列表和设置：使用分隔线或底色，不做浮卡堆叠；
- 用户卡和商品卡：只使用轻边框或浅阴影；
- Modal / Sheet：使用明确遮罩和清楚的表面边界；
- 禁止在装饰卡片内部再套一层装饰卡片。

## 8. 全局组件

### 8.1 头像

- 默认圆形；媒体宫格明确要求时才使用方形；
- 使用 `object-fit: cover` 并保护面部焦点；
- 当前 LTR 构图中，在线薄荷点位于头像左上；
- 头像框、VIP 框、关系框必须使用正式图片素材，不能用 CSS 近似。

### 8.2 用户卡

默认扫描顺序：

1. 头像和在线状态；
2. 用户名、国旗、认证或关系身份；
3. 性别年龄、等级徽章；
4. Bio 或状态；
5. 一个明确的主要社交操作。

用户卡在宫殿渐隐区域上使用白色表面。房间推荐卡可在截图支持时使用粉紫高亮和 Join 操作。

### 8.3 Tab 与筛选

- 一级 Tab 使用文字和紫色下划线；
- 激活文字使用深梅紫或中性深色，并比未激活文字更重；
- 筛选 Chip 保持紧凑，不能抢一级 Tab 的层级；
- 同级互斥选项使用 Segmented Control。

### 8.4 按钮

- 每个区域只允许一个视觉最强操作；
- 通用导航或工具使用熟悉的符号图标；
- 命令不够通用时使用图标加文字；
- 主要社交操作使用正式来源支持的紫、粉紫、蓝或橙金；
- 次操作使用中性表面或轻描边；
- 禁用态降低对比，但不能看起来像已选中；
- 红色只用于真正破坏性的操作。

### 8.5 徽章

- 未读：粉红或红色圆形，白色数值；
- 在线：薄荷圆点或胶囊；
- 性别年龄和等级：保持现有蓝、薄荷、粉色语义家族；
- 奖励和货币：数值必须与正式货币图标成组出现。

### 8.6 Modal 与 Bottom Sheet

- 遮罩必须明确压低底层页面；
- 关闭操作清晰可见，点击区域至少 44 x 44 px；
- Modal 用于短任务、确认和奖励；
- Bottom Sheet 用于浏览、选择、礼物、表情、游戏和工具集合；
- 黑色 Sheet 只用于有当前依据的高对比娱乐目录；
- 关闭后必须恢复底层页面原状态。

### 8.7 空、加载、错误和锁定状态

每个可实现页面必须定义：

- Loading；
- Empty；
- Error / Retry；
- Disabled / Locked（适用时）；
- 长文本和多语言混排；
- Permission Denied（适用时）。

禁止用假数据掩盖缺少空状态的问题。

## 9. 素材规则

### 9.1 只能使用正式素材的内容

- 宫殿背景和五项底部导航插画；
- Laka Logo；
- Coin、Diamond 货币图标；
- VIP 标识和品牌权益素材；
- Relationship / CP 和核心系统徽章。

上述素材缺失时统一标记为 `missing-official`，必须停止并询问，禁止生成替代品。

一次性礼物、临时活动或非品牌装饰实验，在需求允许时可以使用临时生成素材。必须在素材清单中标记 `provisional-ai`，不能意外成为正式母版。

### 9.2 头像

- 优先使用蓝湖正式设计头像；
- 禁止复用或裁取产品截图中的真实用户照片；
- 正式头像缺失时，原型可使用成年、MENA 语境合理的临时头像，并明确标注临时；
- 避免幼态面孔、性化姿势、名人相似脸和文化语境错误的图库图。

### 9.3 文件处理

- 永不覆盖已有素材；
- 按产品用途命名，不按下载时间命名；
- 保留下载到本地的原始文件类型；
- 在 `assets/asset-manifest.md` 中记录来源、分组、页面、日期、状态和允许用途；
- 禁止创建空白 `reference.png` 等占位文件。

## 10. 页面执行流程

每次 UI 任务必须按顺序：

1. 阅读项目 `AGENTS.md`；
2. 阅读本文件；
3. 阅读目标页面 `SCREEN_SPEC.md`；
4. 阅读该页面引用的截图索引；
5. 阅读当前需求中的 UI 意图卡；
6. 分别确认业务来源和视觉来源；
7. 实现前声明哪些区域保持不变；
8. 实现最小且完整的改动；
9. 验证全部状态和未改区域。

禁止为了实现方便而顺手修改其他页面。

### 10.1 修改已有页面

- 从正式版截图和页面规范出发；
- 保留主框架、信息层级、导航、组件身份和无关状态；
- 用最接近的现有模式插入新行为；
- PRD 未明确写出“UI 调整、改版、新风格、重做”时，不得改版。

### 10.2 创建新页面

- 选择最接近的页面家族，复用主框架和组件；
- 开始前确认信息架构和主要状态；
- 产品工具和流程不能做成营销 Landing Page。

### 10.3 原型文件

- 只有当前需求触及页面且要求原型时，才创建或更新 `baseline.html`；
- 禁止创建 `final.html`、`latest.html` 等重复版本；
- HTML、CSS、JS 必须可响应并接近真实产品；
- 需要评审时，加入项目约定的注释评审工具，但不能改变原产品 UI。

## 11. 验收门槛

### 11.1 必须进行视觉对照

在 440 x 956 下，将实现结果与所选正式版截图、当前蓝湖稿（如有）并排比较。

通过条件：

- 结构和信息层级一致；
- 关键位置、颜色、字体、间距和圆角一致；
- 正式素材没有被近似替代；
- 没有重叠、裁切、错误换行和滚动陷阱；
- 阿拉伯语、英语、数字、国旗和徽章可正确共存；
- 未改区域与修改前一致；
- 所有推断都被明确声明。

还需验证一个更窄和一个更宽的常见手机视口。允许自然重排，不允许整屏机械缩放。

### 11.2 状态覆盖

按适用范围验证：

- 默认；
- 加载；
- 空；
- 错误和重试；
- 禁用或锁定；
- Modal / Sheet 打开；
- 键盘打开；
- 权限拒绝；
- 长阿拉伯语和英语；
- 男女角色；
- iOS 和 Android 布局。

### 11.3 Critical Fail

出现以下任一情况，结果直接不通过：

- 伪造或生成正式核心素材；
- 在冲突的正式设计之间静默选择；
- 改动需求未涉及的页面区域；
- 从一级 Tab 移除固定宫殿主题；
- 把真实用户照片作为可复用素材；
- 混淆业务事实和视觉事实优先级；
- 漏掉角色、权限或锁定状态；
- 关键内容重叠、裁切或被遮挡；
- 把未经实际量取的蓝湖尺寸写成精确事实。

## 12. 交付记录

每次 UI 交付必须说明：

- 需求和目标页面；
- 选用的业务来源；
- 选用的视觉来源；
- 修改文件；
- 新增素材及状态；
- 已实现状态；
- 已验证视口；
- 已检查的未改区域；
- 剩余推断或来源冲突。

## 13. 维护规则

- 只有确认的跨页面规则才能更新本文件；
- 页面特有事实更新到 `SCREEN_SPEC.md`；
- 新增正式截图时更新截图索引；
- 新增、替换或升级素材状态时更新素材清单；
- 未经 PM 或正式来源确认，不能把临时推断升级为硬规则。
