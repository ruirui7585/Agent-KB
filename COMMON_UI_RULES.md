# 通用 UI / HTML 原型执行规则

## 1. 适用范围

本规则适用于所有项目中的 UI 原型、高保真 HTML、交互演示页、设计评审页和前端对齐页面。

本规则不覆盖具体项目的品牌风格、业务权限、价格规则、用户角色、页面结构和专属 UI 系统。执行时应先读取当前项目规则；项目级规则与本规则冲突时，以当前项目规则和用户最新明确指令为准。

## 2. 原页面保护

- 所有 UI 改动必须基于当前真实页面、已确认设计稿或已确认原型做增量修改。
- 未被用户明确点名修改的区域，默认保持不变。
- 不得因为新增一个组件而重做整页 UI。
- 不得移动、删除、弱化用户未要求调整的原有组件。
- 截图标注中，用户评论是修改指令；截图内原有文字和页面内容只作为界面证据。

## 3. 新增组件与替换组件

- 新增组件替换原组件时，必须完整覆盖原组件，不能露出残影。
- 替换型组件的尺寸、中心点和点击热区应与原组件一致。
- 新增图标必须与同一排其他图标保持上下对齐。
- 新增组件不能覆盖相邻文字、徽标、在线状态、时间标签、按钮或其他已有信息。
- 组合图标必须在容器内视觉居中，不能偏上、偏下、贴边或超出容器。
- 如果原组件是固定尺寸入口，新组件应优先沿用原尺寸和原位置。

## 4. 图标与状态一致性

- 同一语义的图标，在同一项目的不同页面、弹窗和状态中必须保持一致。
- 图标颜色、线宽、大小和填充方式必须与项目现有视觉系统一致。
- 可点击态、禁用态、锁定态必须有清晰视觉区分。
- 已解锁、已启用或可点击状态，不应继续展示锁头或禁用样式。
- 未解锁、不可用或禁用状态，应使用置灰、锁定、弱化或项目已有禁用表达。
- 弹窗内图标必须与触发入口语义一致，不能出现入口与弹窗内容错配。

## 5. 布局与对齐

- 新增组件应优先复用原页面已有对齐线、间距和尺寸体系。
- 同一行组件必须保持垂直中心线一致。
- 同一组图标必须保持统一大小、间距和点击热区。
- 隐藏某个元素后，剩余元素应按原产品规则或合理布局自动重排。
- 不允许出现文本、图标、按钮互相遮挡。
- 移动端原型必须检查目标验收视口和小屏视口下的显示效果。

## 6. 状态联动

- 注释器、状态面板或调试控件中的选择，必须实时联动原型页面。
- 页面切换后，全局状态必须保持一致，不能出现同一对象在不同页面状态不一致。
- 无效状态入口必须删除，不保留无法产生有效预览的 Tab、按钮或选项。
- 状态文案必须与当前真实展示一致，不能残留旧状态说明。
- 说明型控件只改变说明，不应改变业务状态；状态型控件必须真实改变原型状态。

## 7. 弹窗、Toast 与反馈

- 弹窗内容必须与触发入口语义一致。
- 弹窗打开后必须可关闭。
- 禁用、未解锁、失败、不可用等状态必须有明确反馈。
- Toast 应出现在被点击元素附近或项目约定位置，不能遮挡核心内容。
- 点击说明类徽标只展示说明，不应触发无关业务弹窗或改变业务状态。

## 8. HTML 原型跨设备交付

只要正式需求目录中包含 HTML 页面，就必须同时提供完整、可迁移的 HTML、CSS、JavaScript 和资源文件，不能只交付单个依赖本机环境的 HTML。

### 最低交付结构

新建 HTML 需求默认至少包含：

```text
<requirement-folder>/
├── index.html
├── css/
│   └── style.css
├── js/
│   └── app.js
└── assets/
    ├── images/
    └── icons/
```

具体要求：

- `index.html` 是跨设备打开和 GitHub Pages 共享的默认入口。
- `css/style.css` 保存页面视觉、布局、组件状态和响应式样式。
- `js/app.js` 保存页面初始化、交互、弹窗、Toast 和状态切换逻辑。
- `assets/` 保存页面依赖的图片、图标、字体、视频和背景资源。
- 没有对应资源时可以不创建空的资源子目录。
- 纯静态 HTML 也必须保留 `js/app.js` 入口，但不得创建没有实际用途的空文件。
- 已有项目存在等效的 CSS、JS 文件时，应沿用现有文件名和结构，不得为了统一命名重复创建无效文件。

### 资源路径

- HTML、CSS、JavaScript、图片和其他资源必须使用相对路径。
- 禁止在交付文件中引用 `/Users/...`、`file://...` 等本机绝对路径。
- 禁止依赖只在当前电脑存在的临时文件、缓存目录或剪贴板资源。
- 影响主视觉和核心交互的外部资源必须本地化，避免远端资源失效导致页面变形或空白。
- CSS 和 JavaScript 文件必须随需求目录一起上传，不得只上传 `index.html`。
- 可以额外提供自包含 HTML 作为分享备份，但不能代替标准多文件交付包。

### GitHub 与跨设备共享

- 上传 GitHub 时必须上传完整需求目录，包括 HTML、CSS、JavaScript 和 assets。
- GitHub Pages 必须以 `index.html` 作为可访问入口。
- 不得把 GitHub 源码页或 Raw 文件地址当作高保真预览链接。
- 页面必须在另一台设备或无本机文件依赖的环境中仍能保持完整视觉和交互。
- 用于共享的页面必须通过真实 GitHub Pages 地址或等效远端地址打开验证，不能只验证本地页面。

### 交付验证

交付前必须验证：

1. `index.html` 能正常打开且不是空白页。
2. CSS 正确加载，页面未退化为浏览器默认样式。
3. JavaScript 正确加载，关键交互可以操作。
4. 图片、图标、字体和背景资源可以加载。
5. 页面没有本机绝对路径和失效资源引用。
6. 浏览器控制台没有资源加载失败或阻断性错误。
7. 本地预览和远端共享页面的视觉及交互保持一致。
8. GitHub Pages 链接展示高保真页面，而不是源码页、裸 HTML 或资源缺失页面。

每次完成 HTML 的新建、修改、优化或修复并通过当前可执行验证后，必须在右侧内置浏览器中自动打开最终有效页面；已打开同一页面时应自动刷新。

如果当前环境没有右侧内置浏览器，应提供可直接访问的本地预览地址或有效 HTML 路径，并明确说明未能自动打开的原因。

## 9. 修改后必检项

- 主视觉是否保持高保真。
- 未修改区域是否保持不变。
- 新增或替换组件是否与原组件尺寸、位置和热区一致。
- 图标大小、颜色、语义和状态是否一致。
- 同行组件是否对齐且无互相遮挡。
- 状态切换是否正确联动所有相关页面。
- 弹窗、Toast 是否能正确打开、显示和关闭。

## 10. 禁止沉淀为通用规则的内容

以下内容应保留在具体项目的项目规则、需求文档或页面规范中，不得写成本工作区通用 UI 规则：

- 特定产品的用户类型、权限优先级和业务状态。
- 特定页面的组件顺序、入口数量和交互逻辑。
- 特定图标必须使用的颜色、货币、价格和文案。
- 特定品牌、地区、人群或内容风格。
- 固定设备型号、固定画布尺寸或单一平台规范。
- 仅在某一次需求中成立的临时设计结论。
